Le problème
Les apps Android malmènent la cryptographie

Des études empiriques montrent que la majorité des applications réelles commettent des erreurs critiques dans leurs APIs Java crypto — toutes extractibles depuis un APK en quelques secondes.

CRITIQUE 🔑

Clés hardcodées dans le bytecode

SecretKeySpec("MyK3y!", "AES") est visible en clair après jadx app.apk. N'importe qui peut extraire et déchiffrer toutes les données protégées.

CRITIQUE 🧮

AES-ECB — le mode qui n'en est pas un

Cipher.getInstance("AES") utilise ECB par défaut sur la JVM Android. Blocs identiques → chiffrés identiques. Les patterns de données restent lisibles.

CRITIQUE 🔒

MD5 / SHA-1 — collision en secondes

MD5 est cassé depuis 2004. SHA-1 depuis 2017 (SHAttered). Non conformes NIST SP 800-131A, PCI-DSS, et GDPR Article 32.

MAJEUR 🌐

TLS désactivé, certificats acceptés sans vérification

TrustAllCerts ou hostname verifier vide ouvre la porte aux attaques MITM complètes sur WiFi public — credentials interceptés en clair.

Pipeline
Comment ça marche

De l'APK brut au patch Java corrigé, en une seule requête REST.

01

Soumission — APK, Java ou Kotlin

Envoyez un fichier via l'interface ou POST /api/analyze. Aucun build system requis, aucun accès au code source du projet nécessaire. Formats acceptés : .apk, .java, .kt, ou code brut en JSON.

02

Décompilation APK — Jadx on-the-fly

Pour les APKs, Jadx décompile le bytecode Dalvik en Java source dans un répertoire temporaire isolé. Les librairies tierces (AndroidX, OkHttp, Kotlin stdlib) sont filtrées automatiquement pour concentrer l'analyse sur le code métier de l'application.

03

Analyse statique — 16 règles regex NIST/OWASP

Le moteur applique 16 patterns regex sur le code source concaténé, mappés à NIST SP 800-131A, OWASP MASVS v2, et CWE. Pour chaque match : snippet extrait, numéro de ligne, fenêtre de contexte 5 lignes, sévérité (critique / majeur / mineur).

04

Raisonnement LLM — Qwen2.5-Coder-32B

Les findings sont envoyés à Qwen2.5-Coder-32B via Featherless AI (temperature=0.1). Le modèle produit un score de risque global (0–100), des scénarios d'attaque concrets, l'impact OWASP/GDPR/PCI-DSS, et la priorité de remédiation — en JSON strict validé à la réception. Fallback statique automatique si l'API est indisponible.

05

Génération de patch — code Java corrigé et annoté

Pour chaque finding, POST /api/patch/<rule_id> génère un patch Java complet : imports, gestion d'exceptions, contrainte API level Android, et dépendance Gradle si nécessaire. Les patches ciblent Android Keystore, AES-GCM, et Jetpack Security.

Implémentation IA
LLM au-dessus du statique, pas à la place

Le LLM reçoit des findings déjà validés — il ne détecte pas, il raisonne et génère. Zéro hallucination sur les vulnérabilités.

Pourquoi cette architecture hybride ?

🎯
Précision déterministe

Les règles regex produisent des findings reproductibles et traçables vers les standards NIST/CWE — sans faux positifs LLM.

🧠
Raisonnement contextuel

Qwen2.5-Coder-32B analyse le code source environnant pour construire des scénarios d'attaque réalistes et prioriser la remédiation.

🔧
Patches Android idiomatiques

Les patches générés utilisent Android Keystore System, AES/GCM/NoPadding, et EncryptedSharedPreferences — pas des snippets génériques.

✈️
Mode offline complet

Sans FEATHERLESS_API_KEY, l'outil retourne les findings statiques avec remédiation template. Aucune dépendance externe requise.

Catalogue de règles
16 règles, 7 catégories, 0 config

Chaque règle est mappée à NIST SP 800-131A, OWASP MASVS v2, et un identifiant CWE. Exportable via GET /api/rules.

HASH_MD5 · HASH_SHA1
CRITIQUE

Algorithmes de hachage faibles

MD5 cassé depuis 2004, SHA-1 depuis 2017. Non conformes NIST SP 800-131A.

AES_ECB · AES_CBC_NOAUTH · DES_USAGE
CRITIQUE / MAJEUR

Modes de chiffrement non sécurisés

ECB sans confidentialité sémantique, CBC sans MAC, DES/3DES dépréciés.

HARDCODED_KEY · STATIC_IV
CRITIQUE

Clés et IVs statiques

Extractibles par reverse engineering. Nonce réutilisé avec AES-GCM = récupération de clé.

WEAK_RNG_RANDOM · MATH_RANDOM
MAJEUR

Générateurs pseudo-aléatoires faibles

java.util.Random et Math.random() sont prédictibles. SecureRandom requis en crypto.

SSL_ALL_HOSTS · TRUST_ALL_CERTS · HTTP_CLEAR_TEXT
CRITIQUE / MAJEUR

TLS mal configuré

Hostname verifier vide, TrustManager acceptant tout, trafic HTTP clair → MITM trivial.

SHAREDPREFS_KEY · INTERNAL_STORAGE_KEY
CRITIQUE

Stockage de clés non sécurisé

SharedPreferences non chiffré accessible sur device rooté. World-readable exposé via backup ADB.

Installation
Opérationnel en 3 commandes

Aucun build system, aucune compilation. Python 3.10+ et Java 11+ suffisent. Optionnel : clé Featherless AI pour les features LLM.

Étape 01

Cloner et installer

Clonez le repo et installez les dépendances Python en une commande.

git clone github.com/itsnoha0x/ CryptoLint-AI cd CryptoLint-AI pip install -r requirements.txt
Étape 02

Configurer (optionnel)

Ajoutez votre clé Featherless AI pour activer l'analyse LLM. Sans clé, le mode statique reste complet.

# Créer .env FEATHERLESS_API_KEY=votre_clé # Optionnel : JADX_PATH=/usr/local/bin/jadx
Étape 03

Lancer et auditer

Démarrez le backend et soumettez votre premier APK via l'interface ou l'API REST.

python app.py # → http://localhost:5000 # Ou via curl : curl -X POST localhost:5000/api/analyze -F "file=@app.apk"
